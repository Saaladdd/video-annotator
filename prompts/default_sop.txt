====================================================================
SUBTASK ANNOTATION STANDARD OPERATING PROCEDURE (SOP)
====================================================================

1. PURPOSE
Capture, for each subtask:
- What action is being performed
- Which object is involved
- Where the action occurs (location/context)
- Which hand performs the action (when applicable)
Stay concise and annotate ONLY what is observable and relevant to task execution.

2. ANNOTATION FORMAT
Write each subtask as a short instructional sentence in this order:
    Action + Object + Location/Context + Hand
Examples:
- Pick a screwdriver from the toolbox with the right hand.
- Place the mug on the countertop with the left hand.
- Open the cabinet door with the right hand.
- Pick the basket from the floor with both hands.
Every sentence must be grammatically correct and end with a full stop.

3. ANNOTATION PRINCIPLES
3.1 Observable actions only.
    - Annotate actions that can be directly seen.
    - Do NOT annotate internal thoughts, intentions, or decisions
      (e.g. "Look for a screwdriver", "Decide where to place the mug") unless
      expressed through an observable action.
3.2 Appropriate level of detail.
    - Represent meaningful task steps (e.g. "Pick the screwdriver from the toolbox.").
    - Do NOT over-segment into micro-motions (e.g. "Extend arm", "Close fingers").
3.3 Consistent object names.
    - Use the SAME name for an object throughout the sequence (mug stays "mug", not "cup").
    - Use the GLOBAL VIDEO CONTEXT to keep naming consistent across the clip.
3.4 Ignore unnecessary visual attributes.
    - Do NOT annotate color, size, shape, or appearance
      (e.g. NOT "yellow screwdriver", NOT "blue bottle").
    - EXCEPTION: use a distinguishing attribute only when needed to tell similar
      objects apart (e.g. "Phillips screwdriver" vs "flathead screwdriver").

4. ACTION CATEGORIES
4.1 Manipulation: pick, place, grasp, release, hold, carry, transfer, push, pull,
    rotate, open, close, insert, remove, adjust, align, fold, wipe.
4.2 Navigation (annotate when movement through the environment is required):
    walk, navigate, turn, approach, move away, sit, stand, kneel, bend.
4.3 Task-specific (use when a higher-level action better represents intent):
    tighten screw, loosen screw, apply caulk, paint wall, sow seeds,
    load washing machine, scan barcode, measure board.

5. HAND USAGE
Specify the acting hand when applicable: "with the left hand", "with the right hand",
or "with both hands". If no hand interaction occurs (e.g. "Walk to the kitchen sink."),
do not specify a hand.

6. INTENT AND ACTION START TIME
- A subtask may begin once the intended target is reasonably clear, even before
  physical contact. Do not wait for contact if the target is already obvious.
- If multiple possible targets exist and intent is unclear, delay the start until
  the target becomes identifiable.

7. BODY MOVEMENT
- Annotate gross body movements that contribute to task completion
  (e.g. "Walk to sink.", "Bend toward the basket.", "Kneel beside the planter box.").
- Do NOT annotate eye gaze, head turns, or facial expressions unless essential to the
  task (e.g. "Read measurement on ruler.", "Inspect alignment mark.").

8. TIMESTAMP GUIDELINES
- Report timestamps in SECONDS with 3 decimals (e.g. 12.500).
- Timestamps must be continuous and sequential; avoid unnecessary gaps or overlaps.
- Each line must correspond to ONE distinct observable subtask.
- Use the frame manifest timestamps and GLOBAL VIDEO CONTEXT to set boundaries.
- Never emit a timestamp outside this chunk's range [{chunk_start_timestamp_sec}, {chunk_end_timestamp_sec}].

9. CHECKLIST (apply before answering)
- Action is observable.
- Object is clearly identified and consistently named.
- Level of detail is appropriate (no micro-motions).
- Navigation actions included when required.
- Hand usage specified when applicable.
- Color/size/shape omitted unless disambiguating.
- Timestamps continuous and sequential.
- Every sentence is grammatical and ends with a full stop.

STRICT OUTPUT FORMAT
Output ONLY the header line followed by one subtask per line, in time order.
- Do NOT add prose, commentary, extra sections, or code fences.
- If no observable subtask is present, output exactly: (none)
- Each subtask line MUST follow this pattern:
    [<start_sec>-<end_sec>] <instructional sentence>.
